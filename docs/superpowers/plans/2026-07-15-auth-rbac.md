# Krok 2 — Auth + RBAC (SPEC §4, §8) — implementačný plán

> **For agentic workers:** TDD kde sa dá (RBAC mapovanie, guardy), E2E Auth proti OSTREJ Supabase (lokálna PGlite nemá GoTrue). Zdroj pravdy: SPEC §4/§8 + DECISIONS D9 + zadanie Kroku 2. **Pred písaním klientov znovu over aktuálne @supabase/ssr API** (supabase skill / docs `.md`) — nespoliehať sa na pamäť.

**Cieľ:** Nahradiť stub `src/server/session.ts` reálnym Supabase Auth: login (email+heslo), Proxy chrániaci všetky routes okrem loginu, `getCurrentUser` číta Supabase session (getClaims) + joinuje `users` (rola), RBAC guardy per rola (UI aj server actions), odhlásenie, správa používateľov pre admina.

**Architektúra:** Aplikácia beží dvojkoľajne — (1) **Supabase Auth (GoTrue)** cez `@supabase/ssr` len na overenie identity (session v cookies, JWT); (2) **dáta** cez Drizzle/postgres.js na `DATABASE_URL` (rola v našej `users` tabuľke, NIE v JWT metadata — bezpečnostné pravidlo Supabase). `users.id === auth.users.id` (už v schéme). RLS deny-all na Data API ostáva bezvýznamné pre Drizzle (nepoužívame PostgREST).

**Stack:** `@supabase/ssr` (nainštalovať, pin verziu + lockfile), Next.js 16 App Router **Proxy** (`src/proxy.ts`, nie middleware), server-side auth cez **`getClaims()`** (validuje JWT; NIKDY `getSession()` v server kóde).

## Globálne obmedzenia

- UI po slovensky, terminológia SPEC doslovne.
- **Publishable key ide do prehliadača** (`NEXT_PUBLIC_*`) — service/secret key NIKDY do klienta ani do `NEXT_PUBLIC_`.
- Rola pre autorizáciu VÝHRADNE z DB `users.role` (nie z `user_metadata`/`auth.jwt()` — user-editovateľné).
- Migrácie sa NEMENIA, len pridávajú (žiadna nová sa nečaká — `users`/`user_role` už existujú).
- Ostrá Supabase DB sa nešpiní testovacími dátami; Auth E2E = len prihlásenie existujúceho admina (bezpečné).
- Služby ďalej TDD nad PGlite s mockom session na úrovni `getCurrentUser` (DI vzor).

### Task 4: RBAC mapovanie + guardy (`src/server/rbac.ts`, TDD)

**Files:** Create `src/server/rbac.ts` + `src/server/rbac.test.ts`.

**Produces:**
- `MODUL_ROLE: Record<Modul, UserRole[]>` — mapa modul → povolené roly (SPEC §4): faktúry/dodávatelia/sklad→ekonom+admin; výroba→majster_valcovne+admin; labák→laborant+admin; ciselniky/pouzivatelia→admin; (lisovňa F2).
- `smieVidiet(role, modul): boolean`, `smieVidietRoute(role, pathname): boolean` (mapovanie prefixu routy na modul).
- `overRolu(user, moduly[]): void` — hodí doménovú chybu ak rola nespadá (pre server actions).

- [ ] TDD: každá rola vidí presne svoje moduly; admin všetko; cudzí modul → false/throw. Tabuľkové testy.
